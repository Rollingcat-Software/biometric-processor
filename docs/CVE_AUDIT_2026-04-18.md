# CVE Audit — biometric-processor — 2026-04-18

**Scope:** Read-only inventory of the 79 open Dependabot alerts on
`Rollingcat-Software/biometric-processor` as of 2026-04-18.
No `pip install`, no merges, no commits — upgrade decisions belong to the user.

**Data source:** `gh api /repos/Rollingcat-Software/biometric-processor/dependabot/alerts?state=open --paginate`
(79 records, all severities). Reachability checks via `grep` over `app/`.

---

## 1. Executive summary

1. **79 open alerts cluster into 25 unique packages and roughly 22 distinct upgrade
   actions** (aiohttp alone accounts for 24 alerts but collapses to a single
   `aiohttp>=3.13.4` pin). Severity breakdown: 2 critical, 23 high, 39 moderate,
   15 low.
2. **71 of 79 alerts (90 %) are filed against manifests that are NOT deployed
   to production.** The production image is built from `Dockerfile` →
   `requirements.txt`; Dependabot's hits land on `requirements-lock.txt` (34),
   `requirements-gpu.txt` (37), and `demo-ui/package-lock.json` (3), plus six
   single-hit alternate manifests (`-gcp`, `-railway`, `-render`, `-original`,
   `-laptop-gpu`). The `python-multipart` CVE is the only one that truly
   fans out across every manifest.
3. **Zero alerts are filed against the production `requirements.txt` itself.**
   That file already pins `aiohttp>=3.11.18`, `urllib3>=2.4.0`, `werkzeug>=3.1.3`,
   `jinja2>=3.1.6`, `certifi>=2025.4.26`, `idna>=3.10`, `pillow>=11.2.1`,
   `PyJWT>=2.10.1`, `gunicorn>=23.0.0` — the transitive-security block added
   2026-04-04. The unresolved risk is mainly (a) stale alternate manifests that
   should be deleted or kept in sync and (b) `python-multipart>=0.0.20` which
   still resolves below `0.0.26` at install time.

### Transitive-only (NOT in `requirements.txt`, installed indirectly)

- `keras` (standalone 3.x) — **we use `tf-keras`**. Likely a phantom from an old lockfile. Candidate for deletion.
- `torch` — GPU build only.
- `python-jose` — GPU build only; superseded by PyJWT in prod.
- `orjson`, `msgpack`, `dlib`, `opentelemetry-*` — GPU manifest only.
- `protobuf`, `filelock`, `Jinja2`, `Werkzeug`, `certifi`, `idna`, `urllib3`, `tqdm` — transitive via tensorflow / deepface / requests in lockfile.

Fixing transitive issues that appear only in `requirements-lock.txt` is usually
done by regenerating the lock from an up-to-date `requirements.txt` install,
**not** by adding extra pin lines to `requirements.txt`. Exception:
`python-multipart` is directly declared and should be bumped in place.

## 5. Recommendation

### Bundle A — "safe patch-only PR" (merge in one go)

All of these are pure floor-raises in `requirements.txt` on packages already
there, fixing reachable CVEs with no breaking changes:

```
aiohttp>=3.13.4       # was >=3.11.18
python-multipart>=0.0.26  # was >=0.0.20
PyJWT>=2.12.0         # was >=2.10.1
urllib3>=2.6.3        # was >=2.4.0
werkzeug>=3.1.6       # was >=3.1.3
gdown>=5.2.2          # was >=5.2.0
flask>=3.1.3          # was >=3.1.1
```

Test plan: `docker build` the prod image, run the full pytest suite, smoke
`/healthz`, face enroll, voice enroll, JWT middleware round-trip.

### Bundle B — "maintenance PR" (regenerate stale manifests, no code change)

- Regenerate `requirements-lock.txt` from the current `requirements.txt` install
  so the lockfile stops reporting stale CVEs for `keras 2.15`, `gunicorn 21.2`,
  `urllib3 2.1`, `werkzeug 3.0.1`, `pillow 10.3`, etc.
- Delete or `.dependabot` ignore the unused alternate manifests (`requirements-gpu.txt`,
  `requirements-laptop-gpu.txt`, `requirements-render.txt`, `requirements-railway.txt`,
  `requirements-gcp.txt`, `requirements-original.txt`, `requirements-lock.txt` if no
  longer needed) — these generate ~70 % of the noise.
- Alternatively, add a Dependabot config `open-pull-requests-limit: 0` or
  `ignore` block for those manifest paths.

### Isolated PRs (each one its own test cycle)

1. **`pillow` 11 → 12** — API stable but big version jump; run image upload regression
   (all routes listed in §Reachability: enrollment, verification, quality, comparison,
   landmarks, demographics, multi_face, live_analysis, verification_pipeline, similarity_matrix,
   card_type_router). No reachable CVE vector today (FITS/PSD not accepted), so deprioritize.
2. **`torch` 2.1 → 2.6 (GPU build only)** — if the GPU Dockerfile is ever revived.
   Otherwise recommend deleting `Dockerfile.gpu` + `requirements-gpu.txt`.
3. **`protobuf` 4.25 → 5.29** — requires TensorFlow compatibility check; tensorflow-cpu 2.21
   supports protobuf 5.x but worth a CI dry-run.

### NOT recommended

- Touching `demo-ui/` deps. Not deployed; zero prod impact. Handle on the separate
  web-app/demo cadence if at all.
- Upgrading `keras` standalone — we don't use it. Remove from lockfile instead.
- Replacing `python-jose` — we don't use it in prod. Remove from `requirements-gpu.txt`.
